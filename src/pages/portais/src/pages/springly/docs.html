Visão Geral

Introdução · O que é o Springly

O Springly é uma ferramenta interna do time de Middleware/Operação que automatiza a criação de todos os arquivos de configuração necessários para o deploy de uma aplicação Java SpringBoot em servidores Linux.

Ao invés de criar manualmente cada arquivo — sujeito a erros de digitação e inconsistências entre ambientes — o Springly recebe as informações da aplicação, valida os campos obrigatórios e gera instantaneamente os templates prontos para uso, organizados por categoria e ambiente (Aceite e Produção).

Dica: Todos os arquivos gerados podem ser copiados individualmente com o botão Copiar ou baixados com o botão Download. Use Exportar Todos para baixar todos os arquivos de uma vez.

Como Funciona

Fluxo · Geração de templates

O fluxo do Springly é simples e direto: você preenche um formulário com as informações da aplicação e o sistema substitui automaticamente os valores nos templates pré-definidos, gerando os arquivos corretos para cada ambiente.

1

Clique em "Criar Nova Aplicação"

Na tela principal, acione o botão para abrir o formulário de cadastro da aplicação.

2

Preencha os dados da aplicação

Informe nome, porta, contexto, JAVA_HOME, servidores de aceite/produção, VIPs e o número do chamado. Campos marcados com * são obrigatórios.

3

Clique em "Gerar Configurações"

O sistema valida os campos, gera todos os arquivos e exibe os resultados organizados em abas por categoria.

4

Copie ou baixe os arquivos gerados

Navegue pelas abas (Aceite, Produção, Systemctl, Portal Logs, Apache, Logrotate, Chamados e Comandos) e use os botões de cópia e download para cada arquivo.

5

Aplique nos servidores

Use a aba Comandos como checklist de execução sequencial no servidor Linux. Ela contém todos os comandos de setup com permissões e configurações necessárias.

Requisitos

Pré-requisitos · O que você precisa saber antes

Antes de utilizar o Springly, certifique-se de ter as seguintes informações em mãos:

Acesso aos servidores de aplicação

É necessário acesso SSH aos servidores de Aceite e Produção onde a aplicação será instalada.

Número da porta disponível

Consulte a planilha de cadastro de portas para identificar qual porta será utilizada. Use sempre a subsequente à última porta registrada.

Versão do JDK instalada no servidor

Verifique o caminho exato do JDK no servidor (ex: /opt/jdk/jdk-17.0.8) para preencher o campo JAVA_HOME corretamente.

Endereços VIP de aceite e produção

Obtenha os endereços VIP corretos de cada ambiente junto à equipe de infraestrutura ou no chamado de referência.

Passo a Passo Completo

Guia · Criação de uma aplicação do zero

Abaixo está o fluxo completo de criação de uma aplicação SpringBoot utilizando o Springly, desde o recebimento do chamado até a execução dos comandos no servidor.

Contexto: O contexto da aplicação (ex: /sin/relatorio-lote-habitacional-api) determina em qual grupo de servidores ela será instalada. Analise o chamado para identificar o contexto correto antes de preencher o formulário.

Atenção com os VIPs: Os campos de VIP Aceite e VIP Produção não possuem valor padrão — eles devem ser preenchidos manualmente com o endereço correto de cada ambiente. Nunca copie um VIP de outra aplicação sem confirmar.

Arquivos Gerados

Output · O que o Springly produz

Ao gerar as configurações, o Springly produz os seguintes arquivos, organizados em abas:

start.sh — Aceite

Script de inicialização com profile de aceite e Spring Cloud de aceite

stop.sh — Aceite

Script de parada da aplicação via leitura do pid.txt

start.sh — Produção

Script de inicialização com profile de produção e Spring Cloud de produção

stop.sh — Produção

Script de parada idêntico ao de aceite (path configurado por variável)

springboot-app.service

Unit file do systemd para inicialização automática no boot

Portal Logs PHP (Aceite)

Página PHP do portal de logs apontando para o servidor de aceite

Portal Logs PHP (Produção)

Página PHP do portal de logs apontando para os servidores de produção

httpd.conf — Aceite

Bloco de configuração Apache com Alias e Directory para aceite

httpd.conf — Produção

Bloco Apache com host1 → Servidor Produção 1 e host2 → Servidor Produção 2

Logrotate

Configuração de rotação diária do console.out com compressão

Chamado — VIP

Template de chamado para solicitação de criação das VIPs

Chamado — Monitoração

Template de chamado para solicitação de monitoração da aplicação

start.sh

Referência · Script de inicialização

O start.sh é o script responsável por inicializar a aplicação. Ele configura o JAVA_HOME, define as opções da JVM via JAVA_OPTS e executa a aplicação em background com nohup.

A principal diferença entre o start.sh de Aceite e de Produção são os valores de spring.profiles.active e spring.cloud.config.uri.

start.sh — estrutura
export JAVA_HOME=/opt/jdk/jdk-17.0.8
export PATH=$JAVA_HOME/bin:$PATH

JAVA_OPTS="-Xms128m -Xmx256m -XX:+UseG1GC \
  -Dspring.profiles.active=<profile> \
  -Dspring.cloud.config.uri=<config-uri> \
  -Dgraylog.host=udp:<graylog-host> \
  -Dgraylog.port=12201"

APP_HOME=<caminho-base>/<nome-app>
APP_NAME=<nome-app>.jar

nohup java -server $JAVA_OPTS -jar $APP_HOME/$APP_NAME \
  --server.port=<porta> > $APP_HOME/console.out 2>&1 &

stop.sh

Referência · Script de parada

O stop.sh localiza o processo da aplicação via pid.txt e encerra o processo com kill. Se após 3 segundos o processo ainda estiver ativo, executa um kill -9. O arquivo é idêntico para aceite e produção.

Systemctl

Referência · Unit file do systemd

O arquivo springboot-<nome-app>.service é responsável por configurar a aplicação como um serviço systemd, garantindo que ela seja iniciada automaticamente no boot do servidor.

.service — estrutura
[Unit]
Description=<nome-app>
After=sshd.service

[Service]
Type=forking
User=oracle
WorkingDirectory=<app-path>
ExecStart=<app-path>/start.sh
ExecStop=<app-path>/stop.sh

[Install]
WantedBy=multi-user.target

Após criar o arquivo .service, é obrigatório executar systemctl enable e systemctl daemon-reload, caso contrário a aplicação não iniciará automaticamente no boot.

Apache (httpd.conf)

Referência · Portal de Logs

Os blocos gerados para o httpd.conf criam aliases no Apache que permitem o acesso remoto aos logs da aplicação via o portal de logs interno.

Em Aceite: ambos os aliases (host1 e host2) apontam para o servidor de aceite, pois há apenas uma máquina.
Em Produção: host1 aponta para o Servidor Produção 1 e host2 aponta para o Servidor Produção 2.

Após editar o httpd.conf, reinicie o Apache com systemctl restart httpd.service.

Logrotate

Referência · Rotação de logs

O arquivo de logrotate configura a rotação diária do console.out da aplicação, mantendo as últimas 3 rotações comprimidas na pasta /logs.

logrotate — estrutura
<app-path>/console.out {
    daily
    rotate 3
    missingok
    notifempty
    compress
    dateext
    dateformat -%Y-%m-%d
    olddir <app-path>/logs
    copytruncate
}

Templates de Chamados

Referência · Abertura de chamados

O Springly gera dois templates de chamados prontos para serem abertos no sistema de helpdesk:

Chamado de Criação de VIP

Solicita a criação das VIPs para os ambientes de aceite e produção, com os endpoints e contextos já preenchidos.

Chamado de Monitoração

Solicita a configuração da monitoração da aplicação com servidores, portas, instâncias e VIPs já preenchidos.

Boas Práticas

Recomendações · Uso seguro da ferramenta

Sempre preencha os VIPs manualmente

Os campos de VIP não possuem valor padrão por segurança. Confirme o endereço correto antes de gerar os arquivos.

Consulte a planilha de portas

Verifique sempre a última porta utilizada e use a subsequente para evitar conflitos entre aplicações.

Valide o caminho do JDK no servidor

Execute ls /opt/jdk/ no servidor antes de preencher o JAVA_HOME para garantir que a versão existe.

Use a aba "Comandos" como checklist

Execute os comandos na ordem apresentada na aba Comandos. Não pule etapas, pois as permissões são dependentes umas das outras.

Remova o SNAPSHOT do nome do .jar

Oriente o desenvolvedor a remover sufixos como SNAPSHOT0001 do arquivo .jar antes do deploy, mantendo apenas nome-da-app.jar.
